## Blocker fixed — Savings goals (commit `fix(savings)`)

Root cause: the feature was half-built. `recalculateGoalProgress()` was defined but
**never called**, and `transactions.savings_goal_id` was **never set** by any UI — so a
goal's `current_amount` never reflected money actually moved into savings. This is the
"separo dinero a un saving pero no se relacionan" symptom.

Chosen model: **account-linked derivation (cannot drift).** When a goal is linked to an
account, `current_amount` / `progress` / `is_completed` are computed live in
`getSavingsGoals` from that account's real balance (initial opening + non-deleted
movements). Moving money into the savings account now updates the goal automatically with
zero manual tagging. The goal currency is locked to the account currency in `GoalDialog`.
Goals without an account keep an optional, manually-entered current amount (new field).
Removed the dead `recalculateGoalProgress`.

Files: `src/actions/savings-goals.ts`, `src/app/(dashboard)/savings/_components/GoalDialog.tsx`,
`src/lib/validations/savings-goals.schema.ts`.

How to use: create a savings account, create a goal and pick that account → the card
fills as you transfer money in. Limitation: 1 account ↔ 1 goal (multiple goals on one
account would each show the full balance); for sub-goals use a dedicated account or a
manual goal.

## Other bug fixed — Next-month preview (commit `fix(months)`)

`previewNextMonthFromLatest` summed transaction legs **without** the `deleted_at` filter
that `createMonth` and `recalculateOpeningBalances` both apply, so the projected opening
balances counted soft-deleted transactions. Added the filter.

## Round 2 — deep verification pass (13-agent workflow)

A second pass traced every domain through migrations → RPC SQL → action → hook →
component against real scenarios, plus a full production build and a check that every
referenced RPC/table exists. It confirmed **6 bugs** (fixed in commit `fix: resolve 6
bugs found by full verification pass`):

| Sev | Bug | Location | Fix |
| --- | --- | --- | --- |
| Major | `updateAccount` overwrote every month's opening balance with the same value, breaking the cascade | `accounts.ts` | Set earliest month only, then `recalculateOpeningBalances` |
| Major | `liabilities_year` + `net_worth_evolution_year` dropped debts whose last snapshot was in a prior year | migrations 0013/0015 → `0027` | Carry forward latest snapshot at-or-before the period |
| Major | Investment cost basis not FX-converted in evolution (overstated gains) | `investments.ts` | Apply same rate to cost basis as to current value |
| Major | Debt payment from a different-currency account stored wrong `base_amount` | `debt-activities.ts` | Compute `base_amount` server-side from account currency |
| Major | Recurring weekly/biweekly schedules emitted spurious/misaligned dates | `recurring.ts` | Local-midnight parse, true 14-day cycle, clamp to [start, end] |
| Minor | Forecast cache not invalidated after tx/recurring mutations | `useTransactions.ts`, `useRecurring.ts` | Invalidate `["forecast"]` |

**Downgraded (not real / intentional):** budget multi-leg under-count (SQL has `limit 1`
but the UI only ever creates single-leg income/expense, so unreachable today — left a
note); transfer/debt `exchange_rate` stored from user input while `base_amount` is
recomputed from historical FX (intentional — `base_amount` is authoritative); foreign
currency income/expense into a different-currency account (intentional UI constraint).

**Residual risks left as-is** (documented, not bugs): investment sale-deletion/transfer
restore lots with `purchase_date = sale/transfer date` (only matters if FIFO/tax-lot logic
is added later — current sell uses average cost); investment auto-deduction silently
skipped if no month row exists. Security/RLS verified clean across all tables and RPCs.
